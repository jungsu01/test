function App() {

  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {

    async function fetchData() {
      try {
        const res = await fetch("https://jsonplaceholder.typicode.com/users");

        setUsers(data);

  return (

        ))}

그런데 왜 useRef가 필요하냐?
React는 원래:
DOM 직접 만지는 걸 피함
하지만 가끔 꼭 필요한 경우가 있음.
예:

DOM 직접 조작이 필요한 순간
1) input 자동 포커스
페이지 열자마자 커서 깜빡이게
2) 스크롤 이동
특정 위치로 자동 스크롤
3) 비디오/오디오 제어
play / pause 직접 제어
4) 애니메이션 트리거
DOM 크기 직접 측정

useRef가 하는 역할 : DOM을 "직접 가리키는 손잡이"


예제
const inputRef = useRef(null);
연결
<input ref={inputRef} />
사용
inputRef.current.focus();

inputRef.current.focus();

의미:
React가 만든 input을 직접 찾아서
커서 깜빡이게 함

7. 왜 state로 안 하냐?

state로 하면:
값 변경 → 렌더링 → UI 변화
근데 focus 같은 건:

UI "값"이 아니라 "동작"
이라서 state로 못 처리함.

개념    	의미
state	화면에 표시되는 데이터
ref	화면 요소 자체를 직접 조작
useRef = React가 만든 DOM을 직접 건드릴 수 있는 통로



useState = 화면을 다시 그림 (렌더링 발생)
useRef   = 화면 안 다시 그림 (값만 저장)

구분    	   useState	        useRef
값 변경 시	  화면 다시 렌더링	화면 변화 없음
목적	          UI에 보이는 값	내부 저장 / DOM 접근
사용처	count, input값	DOM, 이전값, 타이머


useState
const [count, setCount] = useState(0);

setCount(count + 1);

👉 화면 바뀜

useRef
const countRef = useRef(0);

countRef.current++;

👉 화면 안 바뀜

핵심 느낌
useState → “보여줄 값”
useRef   → “숨겨둘 값”
2) DOM 직접 조작이 위험한 이유

React는 원래:

React가 DOM을 자동으로 관리
❌ 문제 상황
document.getElementById("box").style.color = "red";

그리고 React state 변경:

setCount(count + 1);
문제 발생 이유

React는:

DOM을 다시 그려버림
결과
내가 바꾼 스타일 → React가 덮어씀
즉
React vs 직접 DOM 조작 = 충돌 발생
그래서 원칙
React에서는 DOM 직접 조작 최소화
→ 꼭 필요할 때만 useRef 사용


useRef 실무 5가지 패턴
① input 자동 포커스
const ref = useRef();

useEffect(() => {
  ref.current.focus();
}, []);
② 이전 값 저장
const prev = useRef();

useEffect(() => {
  prev.current = value;
}, [value]);
③ 타이머 저장
const timer = useRef();

timer.current = setInterval(() => {}, 1000);
④ DOM 크기 측정
const boxRef = useRef();

console.log(boxRef.current.offsetHeight);
⑤ 스크롤 이동
ref.current.scrollIntoView();
4) useRef로 이전 값 기억하기 (중요)

처음 화면
현재: 0
이전: 0
버튼 클릭 (1번)
1. setCount(1)
2. 화면 다시 렌더
3. useEffect 실행
4. prevCount = 1 저장


<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8" />

  <!-- React -->
  <script src="https://unpkg.com/react@18/umd/react.production.min.js"></script>
  <script src="https://unpkg.com/react-dom@18/umd/react-dom.production.min.js"></script>

  <!-- Babel -->
  <script src="https://unpkg.com/@babel/standalone/babel.min.js"></script>
</head>
<body>

<div id="root"></div>

<script type="text/babel">

const { useState, useEffect, useRef } = React;

function App() {

  const [count, setCount] = useState(0);
  const prevCount = useRef(0);

  useEffect(() => {
    prevCount.current = count;
  }, [count]);

  return (
    <div>
      <h1>현재: {count}</h1>
      <h2>이전: {prevCount.current}</h2>

      <button onClick={() => setCount(count + 1)}>
        증가
      </button>
    </div>
  );
}

const root = ReactDOM.createRoot(
  document.getElementById("root")
);

root.render(<App />);

</script>

</body>
</html>

핵심 포인트
왜 이전값이 1개 늦냐?
useEffect는 "렌더 끝난 뒤 실행"이라서
순서
1. count 변경
2. 화면 렌더 (현재 값 먼저 표시)
3. useEffect 실행 (이전값 업데이트)
핵심 한 줄
useRef는 "이전 렌더 값을 저장하는 메모리"


1) setCount 실행 (값 업데이트 예약)
2) render 실행 (화면 다시 그림)
3) 화면에 count 표시 (이미 "새 값")
4) useEffect 실행 (렌더 끝난 뒤)
   → prevCount.current = count 저장


클릭
→ setState 예약
→ render 실행 (이미 최신 count 포함)
→ 화면 표시 (현재값)
→ useEffect 실행 (이전값 저장)

==> 이전값은 "렌더 이후에 저장되는 값"
==> 현재 화면은 render 결과이고, 이전값은 useEffect에서 “그 다음에” 저장된다
화면 = 사진
useEffect = 사진 찍고 나서 기록하는 메모

=> 1 → 0
=> 2 → 1

이렇게 보이는 이유는:
useEffect가 "이전 화면 값"을 다음 렌더에서 보여주기 때문

state가 즉시 안 바뀌는 이유
React는 state를 "즉시 변경"하지 않고 "예약"한다

setCount(count + 1);
console.log(count);

=> 여기서 count는 아직 안 바뀜

왜 바로 안 바꾸냐?
이유 1: 여러 변경을 한 번에 처리
setCount(1)
setCount(2)
setCount(3)

=> React는 다 모아서 한 번에 처리

만약 즉시 바뀐다면?
1 → render
2 → render
3 → render

=> 성능 폭발 (매번 화면 다시 그림)

React 설계 핵심
"여러 state 변경 → 한 번에 render"

setState 여러 번 호출
→ React가 모음
→ render 1번 실행
→ 화면 업데이트

useEffect가 늦게 실행되는 이유
render 끝난 뒤 실행해야 안전하기 때문

만약 render 전에 실행하면:
DOM 아직 없음 → 에러 가능

[구조]
1. state 변경
2. render (UI 계산)
3. 화면 반영
4. useEffect (후처리)

React = 공장
state 변경 = 주문
render = 제품 제작
useEffect = 포장 / 배송 / 라벨 작업

제품 만들기 전에 포장하면 안됨
→ 그래서 useEffect는 마지막

클릭 → state 예약 → render → 화면 → useEffect
React는 "즉시 변경"이 아니라 "모아서 한 번에 업데이트"하는 구조다

dependency 배열이 중요한 이유
useEffect(() => {
  console.log(count);
}, [count]);
의미
count가 바뀔 때만 실행
만약 []이면?
useEffect(() => {}, []);

=> 딱 1번만 실행

만약 없으면?
useEffect(() => {})

=> 렌더 될 때마다 실행 (무한 실행 위험)

핵심
dependency = "언제 다시 실행할지 조건"

state는 즉시 바뀌지 않는다 (예약됨)
2. render는 모아서 1번만 실행된다
3. useEffect는 render 후 실행된다

React = 공장
state = 주문
render = 제품 제작
useEffect = 포장/검수

3. React 방식

React는 직접 DOM을 막 건드리지 않고:

1) Virtual DOM에서 먼저 계산
2) 변경된 부분만 찾음
3) 실제 DOM 최소 수정
4. 흐름 (진짜 중요)

버튼 클릭하면:

1. state 변경
2. Virtual DOM 새로 생성
3. 이전 Virtual DOM과 비교 (Diff)
4. 바뀐 부분만 실제 DOM 수정

처음 화면
<h1>0</h1>

Virtual DOM:

{ count: 0 }
버튼 클릭 (1)

Virtual DOM 새로 생성:

{ count: 1 }
비교 (Diff)
0 → 1 (h1만 바뀜)
실제 DOM 변경
<h1>1</h1>

👉 딱 이것만 수정

9. React가 빠른 이유
1. Virtual DOM에서 먼저 계산
2. 비교해서 최소 변경
3. 실제 DOM 건드리는 횟수 줄임
10. 핵심 한 줄
Virtual DOM은 “실제 화면을 직접 건드리지 않고, 먼저 비교해서 필요한 부분만 바꾸는 중간 단계”다

SPA는 Single Page Application(싱글 페이지 애플리케이션)의 약자이고, React의 핵심 개념 중 하나야.

SPA란?
SPA는 말 그대로 하나의 HTML 페이지로 시작해서, 필요한 부분만 동적으로 바꿔가며 사용하는 웹 앱 구조야.

기존 방식(MPA)과 비교
MPA (Multi Page Application)
페이지 이동할 때마다 서버가 새 HTML을 내려줌
→ 새로고침 발생
SPA (Single Page Application)
처음에 한 번만 HTML을 받고
이후에는 JavaScript가 화면을 바꿔줌
→ 페이지 새로고침 거의 없음
React에서 SPA가 중요한 이유

React는 기본적으로 SPA 구조를 쉽게 만들기 위한 라이브러리야.

어떻게 동작하냐면:
처음 접속 → index.html 한 장만 로드
이후 화면 변경 → 서버 요청 없이 JS가 UI만 변경
필요한 데이터만 API로 가져옴
장점
화면 전환이 빠름 (새로고침 없음)
앱처럼 부드러운 UX
서버 부담 감소 (페이지 전체 재전송 안 함)
단점
처음 로딩이 느릴 수 있음 (JS 파일 큼)
SEO에 추가 대응 필요 (SSR 등)
React랑 SPA 관계 핵심 한 줄

=> React는 “페이지를 새로 받는 대신, 화면을 컴포넌트 단위로 바꿔서 SPA를 구현하는 도구”야.


JSX란?
JSX(JavaScript XML)는 React에서 UI를 쉽게 만들기 위해 사용하는 JavaScript 확장 문법이야.
=> JavaScript 안에서 HTML처럼 UI를 작성할 수 있게 해주는 문법
